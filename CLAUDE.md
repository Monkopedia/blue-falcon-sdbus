# CLAUDE.md

Guidance for AI agents (and humans) working in this repository.

## Project overview

`blue-falcon-sdbus` is a Linux [BlueZ](https://www.bluez.org/) engine for
the [Blue Falcon](https://github.com/Reedyuk/blue-falcon) BLE Kotlin
Multiplatform library. It implements Blue Falcon 3.0's
`dev.bluefalcon.core.BlueFalconEngine` contract on top of
[sdbus-kotlin](https://github.com/Monkopedia/sdbus-kotlin), driving the
BlueZ adapter over the system D-Bus so common BLE code can run on Linux
alongside Android and iOS.

- **Targets:** `linuxX64` and `linuxArm64` (Kotlin/Native).
- **Published to:** Maven Central as `com.monkopedia:blue-falcon-sdbus`.
- **Version scheme:** `<ours>-<blue-falcon-core>` — e.g. `1.0.0-3.0.3`
  means "our 1.0.0 built against `dev.bluefalcon:blue-falcon-core:3.0.3`".
  `version=` lives in `gradle.properties`.

`README.md` is the user-facing API guide; `RELEASING.md` is the release
runbook; `CHANGELOG.md` follows Keep a Changelog.

## Module layout

- **`:engine`** — the engine itself. Hand-written Kotlin lives in
  `engine/src/linuxMain/kotlin/com/monkopedia/bluefalcon/sdbus/`
  (`SdbusEngine` is the bulk of it; plus `SdbusEngineConfig`,
  `SdbusPeripheral`, `SdbusService`, `SdbusCharacteristic`,
  `NoInputNoOutputAgent`). The BlueZ D-Bus interfaces in
  `engine/src/dbus/*.xml` are compiled to proxy/adapter classes
  (`...sdbus.bluez` package) by the sdbus-kotlin Gradle plugin at build
  time — **don't hand-edit generated code; edit the XML and regenerate.**
- **`:integration-tests`** — 14 BLE tests against the
  [BF-Test](https://github.com/Monkopedia/bf-test-peripheral) ESP32-C6
  reference peripheral. Hardware-gated (see below).

## Build, test, and run

Use the Gradle wrapper from the repo root. **Requires JDK 17+** (CI uses
JDK 21) and `libsystemd-dev` (Debian/Ubuntu) / `systemd` libs (Arch) on
the build host — sdbus-kotlin links against `libsystemd`.

- `./gradlew :engine:build` — compiles and links both targets. This is
  the must-be-green gate; CI runs it on every push/PR to `main`.
- `./gradlew :integration-tests:linkDebugTestLinuxX64` — links the
  integration test binary without running it. CI runs this so **API
  breakage is caught even though the tests themselves need hardware.**
- `./gradlew :integration-tests:linuxX64Test -PrunIntegrationTests=true`
  — runs the live BLE suite. Requires hardware (see below). Without the
  `-PrunIntegrationTests=true` flag the test tasks are disabled.

There is no `ktlint`/`spotless`/`apiValidation`/`detekt` configured in
this repo — `:engine:build` is the full check.

## Verification: what an agent CAN and CANNOT confirm

This is the most important rule for autonomous work on this repo.

**Autonomously verifiable (do this before claiming a change works):**

- Compilation of both native targets — `:engine:build`.
- Integration-test binary *links* — `:integration-tests:linkDebugTestLinuxX64`.
- Public API surface / source-level correctness by reading the code.

**NOT autonomously verifiable — needs real hardware:**

- Any live BLE behavior: scan, connect, GATT read/write, notifications,
  bonding. These need an actual BlueZ adapter **and** a peripheral in
  range, so `:integration-tests` cannot be fully exercised by CI or an
  autonomous run.

Consequences for triage/work/review subagents:

- The engine currently has **no unit tests** — `:engine` test tasks
  report `NO-SOURCE`. "Tests pass" here means *compile + link*, not
  behavioral coverage. Don't report a behavioral fix as verified without
  hardware.
- **Do not treat hardware-gated integration-test failures (or their
  absence in CI) as real bugs.** A red live-BLE result on a host with no
  BF-Test peripheral is expected, not a regression.

## Project-specific rules

- **Depends on `sdbus-kotlin`** (`~/git/sdbus-kotlin`, same fleet). This
  engine drives BlueZ entirely through it, and the BlueZ proxies are
  generated by its Gradle plugin. Dependency bumps are **newest-wins and
  coordinated with the sdbus-kotlin agent** — when bumping `sdbusKotlin`
  in `gradle/libs.versions.toml`, watch for version skew and check in
  with that repo's owner rather than pinning unilaterally.
- **Kotlin/Native + cinterop:** changes that touch native interop or the
  D-Bus event loop need careful testing. Native posix / teardown is the
  fleet's known hang area — `SdbusEngine.destroy()` shuts down the D-Bus
  event loop and releases the system bus; mishandling teardown leaks a
  background thread or hangs the process. Treat lifecycle/teardown
  changes as high-risk.
- **Connect retry:** `connect()` deliberately retries the transient
  `org.bluez.Error.Failed: le-connection-abort-by-local` race (3 attempts,
  linear backoff) by default. That's intended behavior, not a bug —
  override via `SdbusEngineConfig.onConnectDelay`, don't remove it.

## Review & merge policy

This is a **public repository**; the review bar leans conservative.

- Changes to the **public API / published surface** (anything in
  `:engine` that affects the `BlueFalconEngine` implementation, the
  `Sdbus*` public types, or the Maven artifact) are **tier-3: a human
  reviews on GitHub. Do not auto-merge these.**
- Docs-only and internal-only changes are lower risk, but still prefer a
  PR through the review pipeline over direct-to-`main`.

## Releasing

Releases are tagged `v<ours>-<core>` (e.g. `v1.0.0-3.0.3`); the tag must
match `gradle.properties` `version=`. The `release.yml` workflow verifies
the tag and a dated `CHANGELOG.md` entry, then publishes to Maven Central
and cuts a GitHub Release. Full checklist — including the
hardware-verified integration run that every release requires — is in
[RELEASING.md](RELEASING.md).
